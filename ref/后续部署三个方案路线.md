## 2. Strategic Route Summary

Recommended long-term route:

1. **Stage 1 — Public Web Release**
   - Goal: make the current local-first Web product publicly deployable under a custom domain.
   - Positioning: privacy-first résumé builder; data remains local unless a later backend is introduced.
   - Output: stable production website with clear deployment docs, CI, metadata, privacy notice, and export smoke tests.

2. **Stage 2 — Web Commercialization / Lightweight App Option**
   - Goal: add monetizable capabilities after Web usage is validated.
   - Preferred monetization first: paid templates, AI generation credits, export enhancements, or account-based cloud sync.
   - Optional package route: PWA first, then Tauri desktop if offline/private positioning is strong.
   - Output: measurable product funnel and one validated packaging or monetization path.

3. **Stage 3 — WeChat Mini Program Channel Expansion**
   - Goal: rebuild for WeChat distribution only after product-market assumptions are clearer.
   - Required architecture: backend login, template service, PDF generation service, payment, storage adapter, and mini-program rendering layer.
   - Output: Taro/WeChat PoC first, not a direct full rewrite.

## 3. Stage 1 — Domain + Web Deployment

### 3.1 Objective

Turn the existing Next.js Web app into a production-ready public website.

The first public release should not require a full backend. The product should explicitly communicate that résumé content is stored locally in the browser unless the user imports/exports files manually.

### 3.2 Recommended Hosting Path

Preferred order:

1. Managed Next.js hosting or static hosting if static export is proven compatible.
2. Node-based deployment if the `/export` print flow or future Next.js behavior makes static export unsuitable.
3. VPS only if there is a hard requirement for China mainland hosting, ICP compliance, custom server control, or backend co-location.

Avoid starting with a self-managed server unless necessary. It creates avoidable operations work: process management, TLS, OS patching, logs, backups, deployment scripts, and security hardening.

- `npm run lint` passes.
- `npm run build` passes.
- Visual tests pass or are updated with explicit review evidence.
- Export/print smoke tests pass against the chosen production build mode.
- CI runs on the real default branch.
- README accurately describes the product and deployment flow.
- The app can be deployed under a domain without introducing a backend.

### 3.5 Main Risks

- Static export may not preserve all runtime assumptions.
- Browser print behavior can vary by browser and platform.
- Resume content is sensitive; public launch needs explicit privacy wording.
- CI branch mismatch can create false confidence if tests never run on the active branch.
- Existing default demo data may confuse first-time users unless UX clarifies it.

## 4. Stage 2 — Web Commercialization and Lightweight App Option

### 4.1 Objective

After the Web app is public and stable, add one commercially meaningful capability without rewriting the product into a new platform.

Recommended priority:

1. Better Web product funnel.
2. Paid templates or export enhancements.
3. AI résumé generation / rewriting.
4. Optional account/cloud sync.
5. PWA.
6. Tauri desktop app only if offline/local-first positioning proves valuable.

### 4.2 Possible Monetization Tracks

#### Track A — Paid templates

Best fit if:

- Users value design quality.
- The app remains mostly local-first.
- Payment can unlock template packs, premium layouts, or export styles.

Architecture impact:

- Template registry.
- Licensing/unlock logic.
- Optional backend for payment verification.
- Anti-abuse does not need to be perfect at early stage.

#### Track B — AI résumé generation credits

Best fit if:

- Users need content generation, optimization, and translation.
- Differentiation comes from AI quality rather than template layout.

Architecture impact:

- Backend required.
- API key must not be exposed in the browser.
- Prompt templates and safety filters required.
- Usage metering required.
- Privacy policy becomes mandatory.

#### Track C — Account and cloud sync

Best fit if:

- Users create multiple resumes.
- Users expect cross-device editing.
- Long-term SaaS retention matters.

Architecture impact:

- Authentication.
- Database.
- Resume CRUD.
- Encryption / privacy policy.
- Account deletion/export.
- Backup/restore.

#### Track D — PWA / Tauri app

Best fit if:

- Users want offline/private editing.
- A desktop app is valuable for trust, privacy, or professional workflow.

Recommended sequence:

1. PWA installability and offline cache.
2. File save/open improvements.
3. Tauri desktop PoC.
4. Native auto-update/signing only after PoC validates export behavior.

### 4.3 Stage 2 Acceptance Criteria

Stage 2 is complete when one monetization or packaging track is validated with:

- A small but working feature slice.
- Explicit privacy/security model.
- Measurable conversion or usage path.
- No regression in PDF/export parity.
- Clear rollback path.

### 4.4 Main Risks

- Adding backend too early can slow the product without proving demand.
- AI features create privacy, cost, and prompt-quality obligations.
- App packaging adds platform QA and distribution overhead.
- Paid templates without traffic do not solve acquisition.

## 5. Stage 3 — WeChat Mini Program Expansion

### 5.1 Objective

Use WeChat mini program as a channel expansion path after the Web product and monetization hypothesis are clearer.

Do not treat Stage 3 as a direct deploy of the existing Next.js app. It is a rewrite/adaptation track.

### 5.2 Required Architecture

Core backend:

- User identity through WeChat login.
- Resume CRUD.
- Template metadata and entitlement.
- Server-side PDF generation.
- Payment verification.
- File delivery.
- Audit logs and order records.

Mini-program frontend:

- Taro 4 + React.
- wx.storage adapter replacing localStorage.
- WeChat page routing.
- WXSS-compatible styling.
- Template rendering adapted to mini-program component constraints.
- Export flow adapted to backend-generated files.

Shared packages:

- Resume TypeScript types.
- Validation schema.
- Template metadata.
- Section ordering logic.
- i18n dictionaries.
- Business rules.

Non-portable or high-risk current Web parts:

- Browser `window.print()`.
- `window.open('/export')`.
- `sessionStorage`.
- `localStorage`.
- Tailwind CSS classes and DOM assumptions.
- Browser file APIs.
- Client-side PDF assumptions.

### 5.3 Recommended Stage 3 Sequence

1. Build backend PDF generation PoC.
2. Build Taro template rendering PoC with one template.
3. Validate wx.storage persistence and import/export substitute.
4. Add WeChat login.
5. Add paid export or paid template test.
6. Expand all templates.
7. Submit mini program review only after payment/security flow is stable.

### 5.4 Stage 3 Acceptance Criteria

Stage 3 is complete when:

- One résumé can be created, edited, persisted, previewed, and exported inside WeChat flow.
- Backend PDF generation output matches Web template quality sufficiently.
- Payment entitlement works and is server-verified.
- WeChat login and data deletion/export policy are documented.
- Mini-program review blockers are resolved.

### 5.5 Main Risks

- Mini program rendering is not pixel-identical to Web.
- PDF generation must move server-side.
- Payment and user data increase compliance burden.
- Rewrite scope can exceed the value of the channel if Web demand is not proven first.
